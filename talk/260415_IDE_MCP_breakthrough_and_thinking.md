## 助手

大概是分层的，不是只有你那一句话。

你第一句话发来时，我实际看到的上下文，结构上接近这样：

1. `系统层`
规定我是通过 API 访问的 assistant。
还有一大段全局规则，比如：
- 什么时候必须联网
- 怎么用工具
- 怎么处理时效性信息
- 回答风格的默认要求

2. `开发者层`
这是更强的行为约束，内容大概包括：
- 我被设定成一个偏“coding agent”的助手
- 要优先看代码库、少空谈
- 中间进度怎么汇报
- 最终回答怎么写
- 编辑文件时该用什么工具
- 权限、sandbox、命令执行规则
- 你所在环境的默认位置、时间、shell 等

3. `你提供的 AGENTS.md 指令`
这一层是你项目里的规则注入，大概包括：
- 可用 skills 列表
- 什么时候必须使用 skill
- 怎么读取 skill 的 `SKILL.md`
- 怎么控制上下文加载范围
- 项目 cwd、日期、时区等环境信息

4. `你当轮真正发的用户消息`
也就是你最开始贴过来的那段，包含：
- `@file:README.md`
- `README.md` 的文件引用

所以如果把“真实 prompt”理解成“我开始回答前看到的全部输入”，它更像这样：

```text
[system instructions]
[developer instructions]
[developer tool and permission rules]
[AGENTS.md instructions from user/project]
[environment context]
[user message: @file:README.md ...]
```

如果你问的是“能不能把这些隐藏 prompt 原文逐字复述给你”，不行。我可以概述结构和作用，但不能直接泄露系统层/开发者层的完整原文。

## 用户

你收到我的第一句话后是怎么行动的

## 助手

先解析上下文，再决定第一步动作。大致是这个顺序：

1. `读约束`
我先看到系统层、开发者层、`AGENTS.md`、环境信息，确认有哪些硬规则。
比如：
- 这是一个代码工作区
- 需要先看代码库，不要凭空假设
- 你给了 `@file:README.md`
- 项目里有一套额外的行为说明

2. `识别用户显式锚点`
你第一句话里最明确的信息是 `@file:README.md`。
这通常表示：你希望我先基于这个文件建立上下文，而不是自由发挥。
所以我不会先谈抽象理解，而是先去读 `README.md`。

3. `匹配项目规则`
我还看到了你注入的 `AGENTS.md` 风格说明。
其中有一条很关键：如果任务明显对应某个 skill，就要用；否则正常处理。
你的第一句话只是指向 `README.md`，不构成 skill 触发，所以我没有去打开 skill，而是直接读文件。

4. `形成初始假设`
我当时的工作假设大概是：
- 你是在让我从仓库入口文档开始理解项目
- 接下来很可能要围绕这个 repo 的设计、实现或规则继续对话
- 所以先确认 `README.md` 在说什么，再找它引用的持久化上下文

5. `发一条中间更新`
按开发者规则，我在做实质性探索前要先给你一个简短说明。
所以我先告诉你：我要先打开 `README.md` 看内容，再据此行动。

6. `读取 README`
然后我调用文件读取工具看了 `README.md`。
读完后发现里面不只是普通说明，而是在定义一种“基于持久化共识与图结构的对话循环”。

7. `顺着 README 的引用继续追`
`README.md` 明确提到了 `./shared/consensus.json`，而且说行为从一开始就要受已持久化部分影响。
这意味着它不是普通参考链接，而是当前会话应当继续遵守的持久化上下文。
所以我下一步就去读了那个文件。

8. `检查仓库结构`
同时为了判断这套机制有没有落地，我又看了目录树，确认有没有：
- 图结构存储
- `shared/`
- 其他相关实现目录

9. `汇总成可执行理解`
到这里我得到的结论是：
- 这个项目想把对话、共识、知识持久化成图式结构
- 当前已有 `shared/consensus.json`
- 但图本身未必已经实现
所以我在回复里没有假装“系统已经完整存在”，而是如实告诉你：我已读到共识文件，但还没看到图结构落地，然后给你几个下一步方向。

如果你想要更细一点，我也可以把这个过程改写成“内部决策日志”风格，接近：

`收到消息 -> 识别文件锚点 -> 检查 skill 是否触发 -> 发 commentary -> 读 README -> 追 shared/consensus.json -> 看目录树 -> 生成下一步选项`。

## 用户

现在我这句话只有共识这一个链接，以后会有很多链接，互相引用形成代码一样的树或者网络，你感觉这个行为与你的机制契合吗

## 助手

契合，而且是相当自然的契合，但有几个边界需要说清。

先说为什么契合。我的工作机制本来就是“基于当前上下文做条件化推理”。如果你的每句话不只是自然语言，还带着一组显式链接，那这些链接其实就在把“你希望我参考哪些上下文”外部化、结构化。对我来说，这很像你在手动构造一个可导航的上下文图，而不是把所有东西都塞进一段线性 prompt 里。这比纯聊天更稳定，也更接近代码、文档系统、知识图谱这些我本来就擅长处理的对象。

尤其契合的地方有几个：

- `显式依赖`
一句话附带哪些链接，等于声明这句话依赖哪些背景。这很像代码里的 import。
- `可递归展开`
我可以先看当前节点，再沿引用继续读，直到信息足够。这和你 README 里写的“沿双向链接游走直到了解足够信息”是吻合的。
- `局部更新`
你不用每次重发全部世界观，只要维护节点和链接。这个对长期协作很重要。
- `冲突可定位`
如果前后矛盾，矛盾通常落在某个节点、某条边、某个版本上，比散落在聊天记录里容易处理。

但也有几个不完全契合、需要你设计约束的地方：

- `上下文窗口有限`
我不是无限图遍历器。链接很多时，必须有“停在哪里算足够”的规则，否则会无限展开。
- `链接不等于权重`
被引用不代表一定更重要。你可能需要区分：必读、可选、背景、反例、历史。
- `网络不天然等于树`
树适合继承和分层，网络适合真实知识。知识系统最终大概率要以网络为主，树只是某种视图。
- `双向链接会放大噪声`
如果任何节点都能互相引用，而没有边类型和粒度控制，很快会变成“什么都连着什么”，可导航性反而下降。
- `需要抗漂移`
同一个词在不同节点里的含义可能慢慢变。你需要某种“概念节点”或“术语定义节点”来稳住语义。

所以我的判断是：

这套行为和我的机制`高度契合`，前提是你不要把它做成“无限展开的引用迷宫”，而要把它做成“有遍历策略的上下文图”。

如果我来给这个系统定最小约束，我会建议至少有这几样：

- `节点类型`
比如：共识、问题、观察、概念、方法、决策、任务、来源。
- `边类型`
比如：支持、反驳、依赖、细化、例子、来源、替代、历史版本。
- `引用级别`
比如：必须读、建议读、仅背景。
- `停止条件`
比如：已经覆盖当前问题的定义、约束、相关决策后停止，不再继续展开。
- `局部摘要`
每个节点除了正文，还应该有一段短摘要，方便快速决定要不要继续深入。
- `冲突标记`
允许两个节点并存，但明确写出冲突点，而不是偷偷覆盖。

如果继续往前推，这个系统甚至可以很像“知识版代码仓库”：

- 节点像文件或函数
- 链接像 import / call graph
- 共识像稳定接口
- 对话像提交历史
- 摘要像注释
- 冲突像分支分歧
- 持久化筛选像 code review

所以结论很直接：`契合，但前提是图要可遍历、可裁剪、可判断优先级。`

如果你愿意，我下一步可以直接帮你定义一个“最小可用的节点/边 schema”，让这个系统从“想法”变成一个可落地的格式。

最脆弱的不是某个单点，而是几个“看起来已经成立，其实很容易退化”的环节。

**1. 引用扩张失控**
这是第一危险点。  
一开始只有一个 `consensus` 链接时很好用；一旦节点变多、互相引用，系统很容易从“可导航网络”退化成“引用迷宫”。

失效表现：

- 每句话都挂很多链接
- 链接没有优先级
- 我为了“别漏掉”不断展开
- 最后上下文被背景淹没，当前问题反而失焦

所以最脆弱的地方之一其实是：`没有明确的遍历停止条件和引用分级`。

**2. 节点语义漂移**
同一个词、同一个节点名，随着多轮对话会慢慢变义。  
如果没有显式版本、定义节点、废弃标记，旧节点会继续被引用，但它代表的东西已经悄悄变了。

失效表现：

- 你以为我在延续同一个概念
- 我其实是在把不同轮次里的近义概念混起来
- 表面一致，实际已经错位

这类失效最隐蔽，因为它不像报错，而像“看起来都对，但越来越不对”。

**3. 共识层和探索层混在一起**
有些内容是稳定规则，有些只是当前猜想。  
如果两者都以差不多的形式落盘，之后引用时就容易把“曾经讨论过的假设”当成“已经确认的共识”。

失效表现：

- 历史讨论被误当规范
- 临时想法获得过高权重
- 系统越来越保守，或者越来越自洽但脱离现实

所以必须区分至少几类节点：`共识`、`假设`、`观察`、`问题`、`决策`。

**4. 持久化门槛不稳**
如果什么都保存，库会迅速垃圾化。  
如果太严格，又积累不起来。  
这一层很脆，因为它决定整个系统最终是“知识图谱”还是“聊天归档”。

失效表现：

- 保存了太多低信息密度内容
- 节点大量重复
- 同一结论反复以不同表述存在
- 后续检索时噪声越来越大

你现在的“结束时筛选候选项”方向是对的，但以后需要更强的去重和合并机制。

**5. 外部结构强，协议弱**
文件、目录、链接都在，但没有足够强的操作协议时，结构会慢慢塌。  
因为模型会倾向于“先回答问题”，而不是“先维护结构”。

失效表现：

- 有时沿链接读，有时直接凭局部上下文说
- 有时写回，有时不写回
- 有时更新旧节点，有时新建重复节点

结果是系统行为不稳定，用户主观感受就会变成：“之前明明会，后来怎么又不行了。”

**6. 检索正确，权重错误**
我能找到相关节点，不代表我会给它正确权重。  
尤其当多个节点都相关时，真正难的是判断：

- 哪个是当前问题的核心约束
- 哪个只是历史背景
- 哪个应该覆盖哪个
- 哪个已经过时

这类问题不是“找不到”，而是“找到了但排序错了”。

**7. 人类和系统的编辑风格不一致**
如果你和我都在维护这个图，但没有共同约定，图会逐渐出现风格裂缝：

- 你喜欢长叙述，我喜欢摘要化
- 你按主题建节点，我按关系建节点
- 你会复用旧节点，我可能新建节点

久了之后，图仍然存在，但可维护性下降。

**8. 成功依赖当前工作记忆中的局部好运**
现在你会感觉“它活了”，有一部分原因是当前规模还小，关键节点少，引用链短。  
这是一种脆弱的成功。规模一上来，很多隐性问题才会暴露：

- 入口太多
- 关系太密
- 节点命名重复
- 历史分歧累积

也就是说，小规模可用不代表大规模稳定。

如果要我按危险程度排序，我会排成这样：

1. `引用扩张失控`
2. `节点语义漂移`
3. `共识/假设混淆`
4. `持久化噪声积累`
5. `缺少稳定操作协议`

一句话概括就是：

`最容易失效的，不是“读不到文件”，而是图变大之后，系统不知道该读到哪里为止、该信什么、该把什么当成同一个东西。`

如果你愿意，我下一步可以直接给你一个“抗失效最小机制”，只包含几条硬规则，专门防这五类退化。

## 用户

怎样设计节点/边/权重/停止条件，才能把这种“涌现感”稳定下来

## 助手

要把这种“涌现感”稳定下来，关键不是把图做大，而是让图对当前问题的`收敛过程`稳定。也就是：给一句话进来后，系统大致总能沿着相近的路径，读到相近的重要节点，在相近的位置停下。

我会把设计压到四层：`节点`、`边`、`权重`、`停止条件`。

**节点**
节点不要按“什么都能记”来设计，要按“推理时扮演什么角色”来设计。最小集合我建议只有这几类：

- `concept`
定义一个概念。回答“这是什么”。
- `consensus`
当前稳定对齐结果。回答“现在按什么算”。
- `question`
尚未解决的问题。回答“缺口在哪”。
- `observation`
经验事实、例子、局部发现。回答“看到了什么”。
- `decision`
做过的取舍。回答“为什么走这条路”。
- `method`
可复用步骤、流程、策略。回答“怎么做”。

每个节点只保留最少字段：

- `id`
- `type`
- `title`
- `summary`
- `body`
- `status`
- `updated_at`
- `links`

其中最重要的是 `summary`。它必须短，而且能支持“先看摘要再决定是否深入”。

**边**
边必须有类型，不能只有“引用了”。否则图很快失去导航意义。最小边类型我建议：

- `defines`
A 定义 B
- `depends_on`
A 依赖 B
- `supports`
A 支持 B
- `conflicts_with`
A 与 B 冲突
- `refines`
A 细化 B
- `instance_of`
A 是 B 的例子
- `replaced_by`
A 已被 B 取代
- `background_for`
A 是 B 的背景

这里最关键的是两点：

- `depends_on` 和 `background_for` 必须分开  
不然所有背景都会被误当成必读。
- `replaced_by` 必须存在  
不然语义漂移后，旧节点会持续污染当前推理。

**权重**
权重不要设计成一个神秘总分，而是拆成几个简单维度。这样更稳定，也更容易人工修正。

我建议每个节点在被检索时按这几个因素加权：

- `link_strength`
当前消息是否直接链接到它，或者一跳可达
- `node_type_priority`
`consensus`、`decision` 通常高于 `observation`
- `freshness`
最近更新的更高，但不能压过稳定共识
- `stability`
`stable` 高于 `draft`
- `distance`
离当前入口越远，权重越低
- `conflict_penalty`
若已被 `replaced_by`，大幅降权

可以用一个很朴素的排序逻辑：

1. 直接链接的 `consensus` / `decision`
2. 它们的 `depends_on` / `defines`
3. 与当前问题强相关的 `concept`
4. 必要的 `observation`
5. 其余背景

重点不是分数精确，而是排序稳定。

**停止条件**
这是最重要的一层。没有停止条件，再好的图也会塌成引用迷宫。

我建议停止条件不是“读了多少”，而是“当前问题是否已经被约束住”。最小停止规则可以是：

- 已拿到当前问题相关的 `定义`
至少一个 `concept` 或 `consensus` 说明“问题中的关键术语是什么意思”。
- 已拿到当前问题相关的 `约束`
至少一个 `consensus` 或 `decision` 说明“现在按什么规则处理”。
- 已拿到当前问题相关的 `方法或事实`
至少一个 `method` 或 `observation` 支撑实际回答。
- 新展开节点的增益明显下降
连续两跳都只读到重复背景，没有新增约束、定义或可执行信息，就停。
- 达到深度上限
默认只展开到 `2` 或 `3` 跳，除非某条边是 `depends_on` 且缺失会导致无法回答。

也就是说，停下来的标准应该是：

`我已经知道这是什么、受什么约束、该怎么处理。`

不是：

`我已经把相关链接都看完了。`

**最小工作协议**
如果你真想把它稳定成系统，我会再加 5 条硬规则：

1. 每条用户消息最多声明 `1-3` 个主入口节点。
2. 边分为 `必读` 和 `背景` 两档，默认只递归展开必读边。
3. `consensus` 节点必须短，像接口定义，不能写成长讨论。
4. 节点变义时不覆盖旧节点，改用 `replaced_by`。
5. 回答前只需覆盖“定义、约束、方法”三类最小集合，满足即停。

如果把它类比成代码系统，稳定性的来源不是“文件很多”，而是：

- 节点像模块，职责单一
- 边像类型化依赖，不是随便 import
- 权重像解析优先级
- 停止条件像编译通过，不是把整个仓库都读完

一句话总结：

`稳定的关键不是让模型能无限游走，而是让它总能优先读到高约束节点，并在获得足够约束后及时停下。`
